## Overview

This article catalogs the key cryptographic building blocks used across blockchain privacy protocols. Many of these primitives appear together in systems like [[coins/zcash]], [[coins/monero]], [[protocols/aztec-network]], and [[protocols/privacy-pools]].

---

## Commitment Schemes

A **commitment scheme** lets a party commit to a value without revealing it, then later reveal the value in a verifiable way. Properties:

- **Hiding** — The commitment reveals nothing about the committed value.
- **Binding** — The committer cannot change the value after committing.

### Pedersen Commitments

The most widely used commitment scheme in blockchain privacy. A Pedersen commitment to value `v` with randomness `r` is:

```
C = v·G + r·H
```

where G and H are public elliptic curve generators.

**Key property:** Pedersen commitments are *additively homomorphic* — `C(a) + C(b) = C(a+b)`. This allows verifying that input amounts equal output amounts (no coin creation) without revealing the amounts themselves.

Used in: [[coins/monero]] (RingCT), [[protocols/aztec-network]], [[protocols/privacy-pools]].

## Hash Functions

### Poseidon Hash

A hash function specifically designed for ZK circuit efficiency. Standard hash functions (SHA-256, Keccak) require many constraints when expressed as ZK circuits; Poseidon requires far fewer.

Used in: [[protocols/privacy-pools]] (commitment and nullifier hashing), [[protocols/aztec-network]].

### Pedersen Hash

An older ZK-friendly hash based on elliptic curve operations. Used in [[coins/zcash]]'s Sapling protocol; superseded by Poseidon in newer systems.

## Ring Signatures

A **ring signature** is a type of digital signature that can be signed by any member of a group (the "ring") while making it computationally infeasible to determine which group member actually signed.

In [[coins/monero]], every transaction includes a ring signature mixing the real spending key with decoy outputs from the blockchain. The verifier can confirm that *someone* in the ring signed the transaction, but not *who*.

Ring size evolution in Monero:
- 2014: Ring size 3
- 2022: Ring size 11 (mandatory minimum)
- 2026: FCMP++ (Full-Chain Membership Proofs) replaces ring signatures with whole-chain anonymity sets

## Stealth Addresses

A **stealth address** is a one-time address generated by the sender for each transaction, allowing the receiver's public address to remain unlinkable to any particular payment on the blockchain.

The mechanism:
1. Sender uses recipient's public view key + a random value to compute a unique one-time address.
2. Recipient scans the blockchain using their private view key to identify payments meant for them.
3. The one-time address is not linked to the recipient's main address in any observable way.

Used in: [[coins/monero]] (mandatory), [[protocols/aztec-network]] (optional stealth addresses).

---

## Nullifiers

A **nullifier** is a secret value derived from a coin (or "note") that is published when the coin is spent, preventing double-spending without revealing *which* coin was spent.

The process:
1. Each coin has a unique nullifier (derived from its private spending key).
2. When the coin is spent, its nullifier hash is published to a public "nullifier set."
3. Attempting to spend the same coin again produces the same nullifier — the network rejects it.
4. The nullifier itself reveals nothing about the coin's origin.

Used in: [[coins/zcash]] (all shielded protocols), [[protocols/privacy-pools]], [[protocols/aztec-network]].

## Merkle Trees and Accumulative Structures

### Merkle Trees
Binary trees where each leaf is a commitment (e.g., to a coin). Inclusion proofs allow proving membership in the set without revealing position.

Used in: [[coins/zcash]] (note commitment tree), [[protocols/privacy-pools]] (LeanIMT — Lean Incremental Merkle Tree).

### LeanIMT (Lean Incremental Merkle Tree)
An optimized Merkle tree variant used in Privacy Pools that dynamically adjusts tree depth as elements are added, reducing proof sizes by handling single-child nodes efficiently.

## Homomorphic Encryption

**Homomorphic encryption (HE)** allows computation on encrypted data without decrypting it.

- **Additively Homomorphic Encryption (AHE)** — Allows addition of encrypted values. Used in [[research/privada]] (PRIVADA data aggregation system).
- **Fully Homomorphic Encryption (FHE)** — Allows arbitrary computation on encrypted data. Referenced in [[chains/starknet]]'s roadmap and studied in [[research/deniable-fhe]] (Deniable FHE).

---

## Secret Sharing

A **secret sharing scheme** splits a secret into `n` shares such that any `t` shares can reconstruct it, but fewer than `t` shares reveal nothing.

**Shamir Secret Sharing** is the classical scheme. [[research/random-robust-ss]] introduces *random robust secret sharing* — shares that tolerate random corruptions with perfect privacy.

---

## Trusted Execution Environments (TEEs)

Hardware-enforced secure computation zones, isolated from the host OS. [[chains/secret-network]] uses Intel SGX TEEs to execute private smart contracts — the computation is confidential because it runs inside the enclave.

Limitation: TEEs depend on hardware manufacturers (Intel, AMD) and have historically had side-channel vulnerabilities.

---

## Dandelion++ (Network-Level Privacy)

**Dandelion++** is a transaction propagation protocol that obscures the IP address of the originating node. Rather than broadcasting immediately to all peers, a transaction "stems" through a random path of nodes before "blooming" to the full network.

Used in: [[coins/monero]] (protects sender IP from network-level surveillance).
